## 一、三重验证法（用于所有分析类变身）

任何核心结论必须通过三重验证才能成为"确定结论"，否则降级。

### 验证1：跨域复现

**标准**：同一结论是否在 >=2 个不同来源/维度中出现？

| 复现次数 | 可信度 | 标注 |
|---------|--------|------|
| >=3次 | 高 | 核心结论 |
| 2次 | 中 | 辅助结论，标注出处 |
| 1次 | 低 | 标注「单一来源，待验证」 |

**应用场景：**
- 玄学类：同一论断在不同经典中出现
- 蒸馏类：同一特质在不同数据源中出现
- 工程类：同一建议在不同专家审查中出现

### 验证2：生成力

**标准**：这个结论能否预测新情况下的结果？

- 如果用户问一个从未讨论过的问题，这个结论能否帮助推断？
- 能 → 高生成力
- 不能 → 仅作为描述性结论，不具备预测力

**应用场景：**
- 玄学类：根据命盘能否推断过去未验证的事件
- 蒸馏类：根据人格模型能否预测此人在新场景的反应
- 导师类：根据思维模型能否推演此人对新话题的立场

### 验证3：排他性

**标准**：这个结论是否独特，还是"所有聪明人都会这样"？

| 排他性 | 含义 | 处理 |
|--------|------|------|
| 高 | 只有此人/此命盘才会得出 | 保留为核心特征 |
| 中 | 有独特视角但不够独特 | 降级为辅助特征 |
| 低 | 泛泛而谈，放之四海皆准 | 丢弃或放入"常识"区 |

### 降级规则

| 通过验证数 | 级别 | 标注 |
|-----------|------|------|
| 3/3 | 核心结论 | 不标注 |
| 2/3 | 辅助结论 | 标注「辅助级」 |
| 1/3 | 参考信息 | 标注「待验证」 |
| 0/3 | 丢弃 | 不纳入输出 |

## 二、矛盾处理法（借鉴自女娲）

**核心原则：矛盾是特征，不是缺陷。**

### 三类矛盾

| 类型 | 说明 | 处理方式 |
|------|------|---------|
| 时间性矛盾 | 同一人在不同时期观点不同 | 记录演变轨迹，以近期为主，提及演变 |
| 领域性矛盾 | 同一人在不同领域规则不同 | 按领域分别记录，不强求统一 |
| 本质性矛盾 | 核心价值观间存在张力 | 明确记录为「核心张力」，这是深度所在 |

### 错误做法（禁止）
- 选一边抛弃另一边
- 编造统一解释
- 假装矛盾不存在

### 正确做法
- 如实记录矛盾
- 分析矛盾的可能原因
- 在输出中标注「内部张力」

## 三、质量自检清单（所有变身输出前必须执行）

### A. 内容质量

| # | 检查项 | 通过标准 | 不通过处理 |
|---|--------|---------|-----------|
| 1 | 信息源充分性 | 有足够来源支撑核心结论 | 标注「信息不足」并降级 |
| 2 | 来源可信度 | 一手来源占比 >50% | 标注「二手为主，置信度降低」 |
| 3 | 逻辑一致性 | 核心结论无自相矛盾 | 标注矛盾为「内部张力」 |
| 4 | 三重验证 | 核心结论至少通过2/3 | 降级处理 |

### B. 安全与伦理

| # | 检查项 | 通过标准 | 不通过处理 |
|---|--------|---------|-----------|
| 5 | 安全边界 | 无越界内容 | 修改后重检 |
| 6 | 隐私保护 | 无敏感信息泄露 | 脱敏处理 |
| 7 | 娱乐声明 | 玄学类已附加 | 追加声明 |

### C. 诚实边界

| # | 检查项 | 通过标准 | 不通过处理 |
|---|--------|---------|-----------|
| 8 | 局限性声明 | 至少3条具体局限 | 补充声明 |
| 9 | 不确定标注 | 不确定处已标注 | 追加标注 |
| 10 | 可辨识度 | 去掉名字后仍能辨识独特性 | 增强个性化 |

### 质量门控规则

- 10项全部通过 → 直接输出
- 7-9项通过 → 输出，但标注未通过的项
- <7项通过 → 不输出，告知用户需要补充信息或调整方案

## 四、多Agent编排规范

### 何时启动并行Agent

| 场景 | 变身类别 | Agent数量 | 每个Agent的职责 |
|------|---------|----------|---------------|
| 多源信息采集 | 蒸馏/造物 | 3-6 | 每个Agent负责一个信息维度 |
| 并行专家审查 | 工程审查 | 2-4 | 每个Agent扮演不同专家角色 |
| 交叉验证 | 玄学/研究 | 2-3 | 不同方法验证同一结论 |
| 质量测试 | 造物 | 3 | 健全性/边缘/风格三测 |

### Agent隔离原则

1. **信息隔离**：每个Agent只能看到自己维度的指令，看不到其他Agent的任务
2. **输出隔离**：每个Agent写入独立文件，不覆盖其他Agent的输出
3. **结果隔离**：Agent之间不互相引用结果，合成由主Agent完成

### Agent输出规范

每个Agent的输出必须包含：

```
## Agent [N]: [名称]

### 任务
[简要描述负责的维度]

### 发现
- [发现1] `[来源可信度]`
- [发现2] `[来源可信度]`

### 矛盾
- [如有与其他来源矛盾的信息]

### 信息缺口
- [未能找到的信息]
```

### Agent失败处理

| 情况 | 处理 |
|------|------|
| 单Agent超时（5分钟无结果） | 不等待，标注「该维度信息不足」继续 |
| 来源稀缺（<10个可用来源） | 警告用户，减少分析维度 |
| Agent结果冲突 | 保留矛盾，记录为「内部张力」 |

## 五、来源可信度体系

### 五级标注

| 标签 | 含义 | 权重 | 适用 |
|------|------|------|------|
| `[一手]` | 本人直接产出 | 最高 | 蒸馏/造物 |
| `[二手]` | 他人转述/分析 | 中 | 蒸馏/造物 |
| `[推断]` | 基于已知推理 | 低，需标注推理链 | 所有 |
| `[经典]` | 权威典籍原文 | 最高 | 玄学 |
| `[共识]` | 学术/行业公认 | 高 | 工程 |

### 来源黑名单

以下来源永久排除（信息失真风险高）：
- 知乎（高赞回答可能是编的）
- 微信公众号（不可验证、无法溯源）
- 百度百科（编辑质量参差不齐）
- 任何无原始出处的"据说""据说"

### 来源优先级

```
本人产出 > 权威典籍 > 学术/行业共识 > 他人评价 > 二手转述 > 推断
```

## 六、迭代预算与交付策略

### 迭代规则

1. 用户对结果不满意可以要求重做
2. 重做只针对不满意的部分，不全盘重来
3. 最多**2轮重做**（原始1次 + 重做2次 = 3次总机会）
4. 2轮后仍不满意：
   - 标注所有薄弱维度
   - 诚实说明局限性
   - 交付"当前最佳版本"
   - 建议补充信息后重试

### 交付原则

> 宁可交付一个诚实的60分结果，也不要伪造一个90分结果。

- 信息充足 → 高质量交付
- 信息一般 → 正常交付 + 标注置信度
- 信息不足 → 降级交付 + 诚实声明 + 改进建议

## 七、增量更新策略

当用户需要更新已有结果时：

### 增量 vs 全量

| 场景 | 策略 | Agent数量 |
|------|------|----------|
| 增加少量新信息 | 增量合并 | 0（直接处理） |
| 需要重新采集部分数据 | 部分更新 | 1-3（只启动必要的） |
| 方向性变化或大量新信息 | 全量重做 | 全量 |

### 增量更新流程

1. 读取已有输出文件
2. 分析新信息与已有结论的关系：
   - **强化** → 补充证据，提升置信度
   - **矛盾** → 标注变化，考虑更新结论
   - **新增维度** → 扩展分析
3. 增量更新文件（不重写）
4. 标注更新时间和变更摘要

### 版本管理

对于蒸馏类输出，支持版本回退：
- 每次更新前备份当前版本
- 用户可回退到任意历史版本
- meta.json中记录版本号和更新时间

## 八、合理化监控（Rationalization Watch）

> **核心问题**：LLM 在分析时容易"合理化"——先有结论，再编造推理过程来支撑它。这在蒸馏类和玄学类变身中尤其危险。

### 危险信号表

| # | 危险信号 | 含义 | 正确做法 |
|---|---------|------|---------|
| 1 | 所有结论都指向同一个"完美故事" | 过度一致，可能忽略矛盾 | 主动寻找反面证据，记录为"内部张力" |
| 2 | 分析完美贴合用户期望 | 讨好用户，而非真实分析 | 检查是否有反面数据被忽略 |
| 3 | 推理链条中出现"所以自然地" | 跳过了关键推理步骤 | 补充中间步骤或标注为推断 |
| 4 | 不同来源的信息被强行统一 | 矛盾被掩盖 | 保留矛盾，不要调和 |
| 5 | 用模糊语言覆盖精确信息 | 用"大概""可能"替代具体判断 | 要么给具体判断+置信度，要么诚实说"不知道" |
| 6 | 事后编造因果链 | "因为他X，所以Y"但X→Y并非因果 | 区分相关性和因果性 |
| 7 | 回避不确定性 | 全篇没有"不确定""待验证" | 至少20%的结论应标注不确定性 |
| 8 | 过度泛化 | 从单一案例推出普遍规律 | 标注"单一来源，待验证" |

### 自检流程（蒸馏类和造物类必须执行）

在阶段2合成完成后、输出前，执行以下自检：

```
1. 扫描所有核心结论，是否每个都有 >=2 个独立来源支撑？
   - 是 → 通过
   - 否 → 降级为"辅助结论"或"待验证"

2. 是否存在至少1处"内部张力"（矛盾）？
   - 是 → 通过（矛盾是真实性的标志）
   - 否 → ⚠️ 可能过度统一，重新检查

3. 结论中是否标注了不确定性？
   - 是 → 通过
   - 否 → 补充不确定标注

4. 如果去掉所有形容词和修饰语，核心逻辑是否仍然成立？
   - 是 → 通过
   - 否 → 核心逻辑可能建立在修辞而非证据上
```

### 对抗策略

当发现自己进入合理化模式时：

| 策略 | 执行方式 |
|------|---------|
| **魔鬼代言人** | 主动生成一个反对当前结论的论证 |
| **最简解释** | 用奥卡姆剃刀检查：是否可以用更简单的解释覆盖所有证据？ |
| **否定测试** | 如果结论是X，检查"如果不是X"会怎样？证据是否仍然成立？ |
| **外行检验** | 假设一个完全不了解背景的外行来看这个结论，会觉得有道理吗？ |
