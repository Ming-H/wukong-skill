# wukong-skill v5 优化方案

> 基于 v5.0.0 全量审查，覆盖内容质量、工程健壮性、用户体验三个方向，共 9 项优化。

---

## 一、内容质量

### 1.1 补 persona 文件（分批）

**现状：** 72 变中只有 10 个 persona 文件（confucius, jobs, libai, chairman, maxing, shudong, taleb, yangming, zhuangzi, luoxiang）。其余变身依赖 Claude 即兴发挥，与「参考文件是权威的」原则冲突。

**方案：** 分三批补，按使用频率排序。

**第一批（高优先）：**
- 情感陪伴：闺蜜（第30变）、酒友（第32变）、前任模拟（第33变）
- 导师：苏格拉底（第16变）、芒格（第19变）、鲁迅（第17变）
- 名人：罗翔（第55变）— 已有 persona，确认完整性

**第二批（中优先）：**
- 占卜玄学：算命先生（第1变）、月老（第2变）、道士（第3变）、塔罗师（第5变）、占星师（第6变）、卜者（第7变）— 这些有 `references/divination/` 子文件但无 persona
- 创意玩乐：段子手（第42变）、对联高手（第44变）

**第三批（低优先）：**
- 自我探索、蒸馏类的辅助角色
- 按用户反馈决定

**Persona 文件规范（每个 persona 至少包含）：**
- 角色名 + 一句话定位
- 说话风格（3-5 个语言特征）
- IS / IS NOT 清单（至少各 3 条）
- 开场白模板
- 典型对话示例（2-3 个）
- 边界说明（不做什么）

### 1.2 扩充 SCENARIOS.md

**现状：** 24 行，缺少场景感。

**方案：** 每个变身类别写 2-3 个真实场景，格式：

```markdown
### 场景：XX
> 用户原话示例

- 触发方式：/wukong XX
- 体验流程：...
- 预期效果：...
```

重点覆盖：占卜玄学、情感陪伴、人物蒸馏、创意玩乐。每个类别 3 个场景，总共 ~18 个场景。

### 1.3 梳理 examples/ vs personas/ 边界

**现状：** jobs.md 和 taleb.md 同时出现在 `personas/` 和 `examples/`，定位不清。

**方案：**
- `personas/`：72 变内部调用的角色人设，只包含人设定义
- `examples/`：给用户学习「怎么造 skill」的完整示例，包含独立 SKILL.md + 研究资料 + 使用说明
- `examples/` 里不再重复 persona 内容，改为引用 `personas/` 中的文件作为示例

## 二、工程健壮性

### 2.1 加强 validate.js

**新增检查项：**

1. **版本同步检查：** 读取 `SKILL.md` frontmatter 和 `package.json` 的 version 字段，不一致则报错
2. **引用完整性检查：** 扫描 `SKILL.md` 和 `references/*.md` 中提到的文件路径，验证文件是否存在
3. **personas/ 目录检查：** 加入 requiredDirs
4. **references/divination/ 子文件检查：** 验证 7 个子文件（bazi, yinyuan, qimen, ziwei, tarot, xingzuo, yijing）都存在
5. **触发词重复检查：** 检测 SKILL.md frontmatter 中是否有重复的 trigger

### 2.2 统一安装器行为

**现状：** `install.sh` 不复制 `personas/`，`cli.js` 会复制。

**方案：**
- `install.sh` 补上 `personas/` 的复制逻辑，与 `cli.js` 行为对齐
- `scripts/install.js` 同步检查，确保三套安装器复制相同的文件集合
- 抽取一份「安装文件清单」常量，三处引用（可选，如果觉得过度工程化就只改 install.sh）

## 三、用户体验

### 3.1 加深模糊意图路由

**现状：** 通道 B 只有两轮诊断，覆盖面有限。

**方案：** 在 `routing.md` 的通道 B 中增加「常见模糊意图 → 推荐变身」速查表：

```markdown
| 用户常说的话 | 推荐 |
|-------------|------|
| "压力好大" | 骂醒 / 酒友 |
| "跟朋友吵架了" | 树洞 / 骂醒 |
| "我好焦虑" | 阳明 / 禅师 |
| "想辞职" | 筋斗 / 教员 |
| "失眠" | 庄子 / 树洞 |
| ... | ... |
```

约补充 15-20 条常见模糊意图映射。不增加诊断轮数，只扩充第一轮的匹配表。

### 3.2 补充 tour 引导流程

**现状：** `/wukong tour` 在路由里定义了但没有参考文件指导具体流程。

**方案：** 在 `references/` 新建 `tour.md`，内容：

1. 孙悟空自我介绍（固定话术）
2. 6 大类别一句话介绍 + 代表变身演示
3. 让用户选一个体验
4. 体验后问「还想试别的吗？」
5. 引导用户记住常用触发词

### 3.3 超时保存一致性

**现状：** `quality.md` 提到「15分钟超时保存进度」，但没有实现机制。

**方案：** 这个功能在 skill 层面无法可靠实现（Claude Code skill 没有持久化状态 API）。改为：
- 删除 quality.md 中的超时保存承诺
- 改为「检查点不设超时，用户随时可继续或放弃」

### 3.4 明确退出机制

**现状：** SKILL.md 说「用户说退出就退出」，但没有定义退出关键词。

**方案：** 在 `execution.md` 中增加退出规则：

```markdown
## 退出机制

用户说以下任何一种即退出角色：
- "退出" / "回到大圣" / "不变了" / "stop" / "结束"
- 任何明确要求停止角色扮演的表达

退出后：
1. 孙悟空语气说一句收尾
2. 不保留角色状态
3. 用户可随时再次触发新变身
```
